### 1. Code review (multi-angle, verified)

Ran 8 parallel finder agents (line-by-line scan, removed-behavior audit,
cross-file trace, reuse, simplification, efficiency, altitude, conventions)
over `git diff main...HEAD`, then verified every candidate with a second
independent agent pass. 13 candidates were refuted as deliberate/documented
design (e.g. the `answer_hash` session-based dedup, strict token-line
grammar, feedback-key strictness — all have comments or tests asserting
they're intentional). 7 real issues survived verification and were fixed:

1. **Rate-limit quota burned on failed writes** — `app/api/submissions/route.ts`
   reserved the daily slot before `database.batch()`; a transient D1 failure
   left the slot consumed with nothing stored. Fixed by releasing the slot
   (`releaseDailySubmission`, new in `lib/antiAbuse.server.ts`) if the batch
   insert throws.
2. **Honeypot response shape mismatch** — the honeypot path returned a bare
   `{ok:true}` while the real client success handler
   (`app/BenchmarkApp.tsx`) throws unless `score`/`maxScore`/
   `eligibleForPrimaryAnalysis` are present. Fixed by returning the full
   payload shape (zeroed) from the honeypot branch.
3. **Locale-dependent scoring** — `lib/scoring.ts` used `.toLocaleLowerCase()`
   / `.toLocaleUpperCase()` in `normalizeText`, `parsePosition`, the
   before/after ordering check, and `normalizeFieldName`. Under a
   Turkish-locale runtime, `"I".toLocaleLowerCase()` → `"ı"` (dotless),
   which would silently zero otherwise-correct answers. Replaced all with
   locale-invariant `.toLowerCase()`/`.toUpperCase()`.
4. **Double config validation per submission** — `scoreResponses` re-ran
   `parseScoringConfig` on a config `loadScoringConfig` had already parsed.
   Added a `WeakSet` of already-validated config objects so re-parsing is
   skipped.
5. **Regex recompiled every scoring call** — `scoreRegex`/`scoreRegexSet`
   built a fresh `RegExp` per rule per submission. Added a module-level
   `REGEX_CACHE` keyed by `flags+pattern`, resetting `lastIndex` on reuse
   (relevant for sticky/global flags).
6. **Static scoring rules refetched from D1 every POST** — `loadScoringConfig`
   in `lib/scoringConfig.server.ts` had no caching. Added a `WeakMap`-keyed
   (per D1 binding instance) cache with a 5-minute TTL, so test fixtures with
   fresh fake databases are unaffected.
7. **Duplicated 249-entry ISO country list** — `lib/submission.ts` had its own
   copy of the same list already exported as `COUNTRY_CODES` from
   `lib/countries.ts`. Now imports it. Also added a CI test
   (`tests/submission.test.ts`) asserting `BENCHMARK_PROMPTS.length ===
   EXPECTED_PROMPT_COUNT`, because `assertRequiredPromptIds` throws a raw
   `RangeError` (not `SubmissionValidationError`) on a count mismatch, which
   would escape the route's error wrapper and turn every POST into a 500 —
   this locks the two constants together so a future drift fails tests
   instead of production.
